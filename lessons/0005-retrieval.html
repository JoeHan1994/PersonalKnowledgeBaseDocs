Lesson 0005 · 个人知识库

检索:把问题变成一段"带出处的提示词"

这是在线的那一半——嵌问题、取 top-k、拼上下文。

这一课的小胜利:你能输入一个问题,亲眼看着系统给候选块打分排序、 取出最相关的 k 个,并实时拼成一段带 [1][2] 出处的提示词; 然后说清这套编排为什么属于你项目的 application 层。

先认词 · 本课会反复出现

检索 Retrieval
在线阶段:把问题也嵌成向量,去向量库里取出最相关的几块原文。RAG 里的"查资料"。
相似度分数 Score
每个候选块和问题有多像的数值。越高越靠前,排序就靠它。
上下文 / 提示词 Context / Prompt
把取回的原文 + 问题拼成的一段话,交给模型照着答。本课的产物。
出处标注 Citation
给每块资料编号 [1][2],让答案能指回原文,可追溯、可信。
重排 Re-rank
先粗筛一批,再用更精的模型把"最相关的"重新排到最前。可选的进阶环节。
用例 / 应用层 Use case / application
把多步操作"编排"成一个业务动作的地方。只调端口,不碰具体实现。

检索做了哪几步?

回忆在线管道:[L1] 问题嵌入 → 检索 top-k → 拼接 → 生成。这一课聚焦中间三步(生成留到下一课):

· 嵌问题:用和建库时同一个嵌入模型,把问题也变成向量 [L3]。
· 取 top-k:向量库返回最相似的 k 个块,每个带分数和元数据 [L4]。
· 拼上下文:把这 k 段原文编号,和问题拼成一段带出处的提示词。

亲手试:输入问题 → 看它排序 + 拼装

下面是一个迷你知识库(几条已入库的块)。输入问题,系统给每块打分、取 top-k,并在下方实时拼出提示词。 (这里用"词语重合"当相似度的简化替身;真实里换成余弦,思路一样 👇)

拼出来的提示词长什么样

核心模板就一句:"只根据下面的资料回答,并标注出处编号;答不出就说不知道。" 后面跟编号的资料,再跟问题。"答不出就说不知道"这句很关键——它把模型摁在资料里, 少编造。

落到你的项目

检索是一段编排:嵌问题、查库、拼提示。它只调用端口 (LLMProvider.embeddings、VectorStore.query),不关心谁来实现—— 所以它是一个 application 用例:

# app/application/retrieve_context.py(应用层 · 只编排,不碰 I/O 细节)
class RetrieveContext:
    def __init__(self, embedder: LLMProvider, store: VectorStore): ...
    async def run(self, question: str, k: int) -> list[Chunk]:
        vec = (await self.embedder.embeddings(MODEL, [question]))[0]
        return await self.store.query(vec, k)  # top-k 块

记住:"嵌问题 + 查库 + 拼上下文"是 application 的编排; 具体怎么嵌、怎么查,是 infrastructure 的事。 下一课会接上最后一步——把这段提示交给已有的 LLMProvider,生成带出处的回答, 打通你的第一个垂直切片。

进阶一眼:重排(re-rank)

当 top-k 还不够准,可加两段式:先用向量粗筛一批(如 top-20), 再用更精细的模型(cross-encoder)给这 20 个重新打分,只留最好的 3 个。[1] 先知道有这招即可,主线打通后再加。
