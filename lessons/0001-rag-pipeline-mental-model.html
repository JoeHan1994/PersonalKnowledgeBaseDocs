先认词 · 本课会反复出现

RAG 检索增强生成
先查资料再答题。不靠模型死记,而是临场翻出相关原文喂给它,让它照着答。本课主角。
嵌入 / 向量 Embedding / Vector
把一段文字变成一串数字(坐标)。意思相近的文字,坐标也相近——于是"找相似"变成"找邻居"。
向量库 Vector Store
专门存这些数字坐标、并能飞快"找最近邻"的数据库。资料的家。
检索 / top-k Retrieval
从向量库里挑出离问题最近的 k 段原文(比如前 3 段),当作答题的"参考资料"。
管道 · 离线/在线 Pipeline
一串按顺序的加工步骤。离线=平时建库(资料多了才跑);在线=每次提问都跑。
出处 Source / Citation
答案背后引用的那段原文。能给出处 = 可追溯、可信。
端口 / 适配器 Port / Adapter
端口="我需要一个能干这事的东西"的约定(接口);适配器=真正干活的具体实现。换实现不动主逻辑。

为什么不直接把资料塞给模型?

大语言模型把知识压进了自己的权重里(参数化记忆),但它没法精确地 访问、更新或给出处。对私人资料,它干脆没见过。 RAG 的核心主张是:给模型再配一块非参数化记忆——一个可检索的向量索引—— 回答时先检索出相关原文,再让模型基于检索到的内容作答。[1]

这一步换来三样你正好需要的东西:可给出处、可更新(加资料不用重训模型)、 以及隐私可控(资料留在你的本地索引里)。

管道分两个阶段

关键洞察:RAG 不是一条线,而是两条——一条离线建库,一条在线问答。它们在"向量库"这里交汇。

离线 · INGESTION(建库,资料变多时才跑)

加载读取原始资料
Loader
分块切成小段
Chunk
嵌入每段算向量
Embedding
入库存向量+元数据
Vector Store

在线 · QUERY(每次提问都跑)

问题嵌入把提问也算成向量
检索取最相似的 top-k 段
拼接原文+问题组成提示
生成模型据此带出处作答

映射到你的架构(这才是重点)

你的项目是 Onion/DDD,依赖向内。新建知识库能力时,每一步该放哪层是确定的:

· domain 纯数据与规则:Document、Chunk 实体, VectorStore / Embedder 端口(port)。无 I/O、无框架。
· application 用例编排:IngestDocument(加载→分块→嵌入→入库)、 AnswerQuestion(检索→拼接→调用已有 LLMProvider)。
· infrastructure 适配器:PDF/Markdown Loader、 Chroma 向量库实现、Ollama 嵌入实现。唯一允许碰 I/O 的地方。
· interface FastAPI 路由:POST /knowledge/ingest、POST /knowledge/ask。

好消息:嵌入这一步你已经有港口了——LLMProvider.embeddings() 早已定义。 所以你缺的主要是 Document/Chunk 模型、VectorStore 端口,以及两个用例。
