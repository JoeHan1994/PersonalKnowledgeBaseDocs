Lesson 0002 · 个人知识库

分块:把长文切成可检索的小段

检索的质量,从这一步就被决定了。

这一课的小胜利:你能亲手调 chunk_size 和 overlap 两个旋钮、 实时看到一段文字被怎样切开,并说清"为什么要重叠"——然后把这套规则写成你项目里 domain 的 Chunk 概念。

先认词 · 本课会反复出现

分块 / 块 Chunk
把长文章切成的一小段。它是知识库里检索和引用的最小单位——以后查出来、给出处,都是按"块"。
chunk_size 块大小
每一块多长(按字符数算)。太大→一块混太多主题;太小→句子被切碎读不懂。
overlap 重叠
相邻两块之间共享的一小段。防止"一句话被劈成两半,谁也读不全"。常取块大小的 10%–20%。
元数据 Metadata
跟着每块一起存的"标签信息":来源文件、页码、标题路径等。检索后用来标出处、做过滤。
领域层 domain
项目里放纯数据和规则的最内层,不碰文件/网络。Chunk 这种纯数据就住这儿。

为什么必须分块?

回忆上一课的离线管道:[L1] 加载 → 分块 → 嵌入 → 入库。为什么不整篇文章算一个向量?三个原因:

· 嵌入有上限:嵌入模型一次能"读"的文本有限,太长会被截断,信息丢失。
· 检索要精准:你问一个细节,系统应只取出相关的那一小段当出处,而不是整篇噪声。
· 上下文要省:塞给生成模型的内容越少越准、越快、越省。

所以"块"是知识库里检索和引用的最小单位。块切得好,后面全顺;切得烂,嵌入再强也救不回。

两个旋钮:大小与重叠

最常用的策略叫固定大小+重叠:按长度切块,相邻块之间留一段重叠。 重叠是为了防止"一句话被劈成两半,谁也读不懂"——让跨边界的语义在两块里都各留一份。[1] 亲手试试 👇

怎么选这两个数?

没有唯一正确值,但有经验区间:chunk_size 常取几百字符(对应一两段话); overlap 常取 chunk_size 的 10%–20%。 把上面滑块拉到极端,你会直观感到两端的代价:

· 块太大 → 一个向量背负多个主题,检索"什么都沾一点、什么都不准"。
· 块太小 → 句子被劈碎,单块缺乏上下文,模型读不懂。
· 零重叠 → 跨边界的关键句两边都不完整,检索可能两块都漏。

落到你的项目

分块产出的"块"是纯数据,没有任何 I/O——所以它属于 domain:

# app/domain/models.py(领域层 · 纯数据)
@dataclass(frozen=True)
class Chunk:
    document_id: str
    index: int    # 第几块,用于还原顺序
    text: str
    # 之后会加 embedding + 出处元数据

而"怎么切"这套算法,会在第 4–6 课接进来时,作为一个可替换的策略放到 application / infrastructure—— 因为不同数据源(代码 vs 散文)可能需要不同切法。先记住:数据(Chunk)在内层,算法在外层。
