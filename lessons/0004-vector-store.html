Lesson 0004 · 个人知识库

向量库:让"找最近的几个"快如闪电

存的是"块 + 向量 + 元数据";取的是"离问题最近的 top-k"。

这一课的小胜利:你能在画布上点一个"问题",亲眼看着向量库 挑出离它最近的 k 个块并连线;然后说清 Chroma 存什么、 query 返回什么,并把它落成你项目 infrastructure 的 VectorStore 适配器。

先认词 · 本课会反复出现

向量库 Vector Store / DB
专门存"向量 + 原文 + 元数据",并能飞快找最近邻的数据库。本课用 Chroma。
最近邻 / top-k Nearest Neighbours
离问题向量"最近"的那几个块。取前 k 个(如 top-3)当答题的参考资料。
ANN 近似最近邻 Approximate NN
不逐个死算,而是用聪明的索引快速猜出最近的几个。牺牲一丁点精度,换巨大速度。
HNSW 分层小世界图
一种主流 ANN 索引。把向量连成"多层高速公路图",查询时先上高层快速逼近,再下层细找。
集合 Collection
Chroma 里装同一批数据的"表/容器"。建集合时顺便选用哪种距离度量(l2/cosine/ip)。
元数据 Metadata
跟每块一起存的标签(来源、页码…),检索时可据此过滤和标出处。
端口 / 适配器 Port / Adapter
端口=领域层声明"我要个能存能查的向量库";适配器=基础层用 Chroma 真正实现它。换库不动主逻辑。

向量库到底是什么?

回忆离线管道:[L1] 加载 → 分块 [L2] → 嵌入 [L3] → 入库。 向量库就是这个"库"——它替你存三样东西、并把最近邻检索做到极快:

· 向量:每个块的嵌入(那串浮点数)。
· 原文:块的文本本身(检索回来要当出处喂给模型)。
· 元数据:来源、页码、标签……用于过滤和标注出处。

检索 = 在向量空间里找最近的几个

为什么能"秒级"?近似最近邻(ANN)

库里几十万、几百万个向量,逐个算距离太慢。真正的向量库用 HNSW 这类近似最近邻(ANN)索引—— 把向量组织成"多层高速公路图",查询时从高层快速跳到目标附近,再逐层细找。[1]

代价:它是近似的,极偶尔会漏掉真正的第一名。 回报:在海量向量下依然毫秒级。对知识库检索,这笔交易几乎总是划算的。

Chroma 长什么样

我们已选定 Chroma [L3]。它的接口就两个核心动作—— 加(add) 和 查(query):

collection = client.create_collection(
    name="kb",
    configuration={"hnsw": {"space": "cosine"}},  # l2(默认)/ cosine / ip
)
collection.add(    # 存:向量 + 原文 + 元数据 + id
    ids=["c1"], embeddings=[[...]],
    documents=["这一块的原文"], metadatas=[{"source": "note.md", "page": 1}],
)
res = collection.query( # 查:给问题向量,要回最近的 k 个
    query_embeddings=[[...]], n_results=3,
)  # res 里有 documents / metadatas / distances

落到你的项目

依赖向内:domain 只声明要一个能存能查的库(端口), 至于"是 Chroma 还是别的",由 infrastructure 的适配器决定。 这样将来换库不动核心逻辑:

# app/domain/ports.py(领域层 · 只声明契约)
class VectorStore(Protocol):
    async def add(self, chunks: list[Chunk], vectors: list[list[float]]) -> None: ...
    async def query(self, vector: list[float], k: int) -> list[Chunk]: ...

# app/infrastructure/vectorstore/chroma_store.py(基础层 · 真正调 Chroma)
class ChromaVectorStore:  # 实现上面的契约
    ...

记住分界:"要一个向量库"是 domain 的接口; "用 Chroma 实现"是 infrastructure 的适配器。 下一课会用它做检索:把问题嵌入、查 top-k、组装成"带出处的上下文"。
