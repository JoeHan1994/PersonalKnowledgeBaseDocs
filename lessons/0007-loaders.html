加载器:把"乱七八糟的格式"收成统一文档

这是你使命的正面战场——异构数据,统一入口。

这一课的小胜利:你能粘一段带格式的 Markdown,看着加载器把它规整成一个 统一的 Document(抽出元数据、剥掉语法噪声、附上来源); 并说清"为什么每加一种新格式,只需加一个 infrastructure 适配器"。

先认词 · 本课会反复出现

加载器 Loader
读一个来源(文件/网页/…)、把它解析成统一文档的部件。异构数据进、统一格式出。本课主角。
文档 Document
加载后的标准产物:{ 正文文本 + 元数据 }。之后交给分块器切成块(第 2 课)。
异构数据 Heterogeneous
格式五花八门的资料:Markdown、PDF、网页、代码、聊天记录……你最初的痛点。
前置元信息 Front-matter
Markdown 顶部 --- 之间的键值(标题/标签/日期)。加载时抽出来当元数据。
归一化 Normalize
把不同格式"洗"成同一种干净结构,去掉语法噪声,只留正文 + 有用标签。
注册表 / 工厂 Registry / Factory
按文件类型挑对应加载器的小调度表。加新格式 = 往表里加一行。

为什么需要加载器?

回忆离线管道第一步:[L1] 加载 → 分块 → 嵌入 → 入库。前几课我们用的是"示例数据"; 现在要喂真实资料,可它们格式各异——这正是你立项时说的"数据源很乱"。

加载器的使命就一句话:不管进来的是什么格式,出去的都是同一种 Document。 后面的分块、嵌入、入库完全不需要知道资料原本是 PDF 还是网页—— 复杂性被挡在了加载这一层。

嵌入每块→向量
入库存进向量库

同一个出口,多种入口

Markdown 只是第一个。其它格式各写一个加载器,但产物形状完全一样—— 这就是它们能共用后续整条管道的原因:

输入格式加载器要做的脏活统一产物
Markdown / 文本抽 front-matter、剥语法Document
{ text, metadata }
PDF按页抽文字(pypdf)
网页 HTML取正文、扔掉导航/脚本
代码仓库按文件/函数读,记符号
聊天 / 邮件按对话/时间聚合

这些"按类型切"的细节,你在 分块速查表 的可运行脚本里已经见过——加载器负责读取 + 归一化,分块器负责切分,两者接力。

落到你的项目

"要一个能把来源变成 Document 的东西"是契约,放 domain; "用 pypdf / HTMLParser 真去读"是 I/O,放 infrastructure; "按扩展名挑哪个加载器"是一张小注册表:

# app/domain/ports.py(领域层 · 契约)
class Loader(Protocol):
    def load(self, path: str) -> Document: ...

# app/infrastructure/loaders/markdown_loader.py(基础层 · 真读文件)
class MarkdownLoader:  # 实现 Loader
    def load(self, path): ...  # 抽 front-matter + 剥语法

# 注册表:扩展名 → 加载器(加新格式只动这里)
LOADERS = {".md": MarkdownLoader(), ".txt": TextLoader()}

记住:加新数据源 = 写一个新适配器 + 往注册表加一行,核心管道一行不改。 这就是为什么"数据源很乱"不再可怕——乱被关进了加载层。
